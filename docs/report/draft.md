## 1. Ralph: Bash 한 줄의 철학

### 왜 이게 탄생했나

Ralph를 처음 봤을 때 솔직히 당황했다. 핵심 파일인 `ralph.sh`를 열어보면 실제로 이게 전부다:

```bash
while :; do cat PROMPT.md | claude-code; done
```

14,000개가 넘는 GitHub 스타를 받은 프로젝트의 핵심이 Bash 무한 루프 한 줄이라는 게 처음엔 이해가 안 됐다. 그런데 원작자 Geoffrey Huntley의 글을 읽고 나서 의도가 보였다.

그는 개발자들이 AI 코딩 도구를 쓸 때 겪는 핵심 문제를 이렇게 진단했다: **컨텍스트 윈도우가 한계에 달하면 AI가 망가진다.** 대화가 길어질수록 AI는 앞에서 한 결정을 잊어버리고, 일관성이 무너지고, 에러가 누적된다. 그래서 그가 선택한 해법은 역설적이게도 "매번 처음부터 시작하자"는 것이었다.

### 구조를 뜯어보니

Ralph의 파일 구조는 단순하지만 영리하게 설계되어 있다.

- **`prd.json`**: 유저 스토리 목록. 각 스토리마다 `passes: true/false` 상태를 갖는다.
- **`progress.txt`**: Append-only 학습 기록. 이전 반복에서 배운 것들이 누적된다.
- **`AGENTS.md`**: AI가 발견한 재사용 가능한 패턴을 스스로 기록하는 파일.

매 반복이 새로운 AI 인스턴스로 시작되지만, 이 세 파일을 통해 이전 반복의 지식이 전달된다. 상태는 AI의 기억이 아니라 파일 시스템에 저장된다는 발상이다. 분산 시스템의 이벤트 소싱 패턴을 개발 워크플로우에 적용한 것처럼 보인다.

워크플로우는 이렇다:

```
1. PRD 작성 → prd.json 생성
2. ralph.sh 실행
3. AI가 passes=false인 스토리 하나 선택 → 구현 → typecheck/lint/test
4. 통과하면 passes=true, progress.txt에 학습 내용 추가
5. 모든 스토리 완료 → 루프 종료
```

### 흥미로웠던 점

Huntley가 직접 밝힌 사례가 인상적이었다. "$50,000짜리 계약 규모의 MVP를 $297에 완성했다"고 했는데, 이게 과장이더라도 방향성은 분명하다: AI를 "계속 감독해야 하는 도구"가 아니라 "혼자 돌아가는 공장"처럼 쓴다는 것.

다만 원작자 스스로 "레거시 코드베이스에는 쓰지 않겠다"고 밝혔다. 매 반복이 새로운 컨텍스트이기 때문에, 기존 코드의 맥락을 깊게 이해해야 하는 작업엔 취약하다.

### 언제 쓸까

새 프로젝트를 처음부터 만들 때, 그리고 PRD를 명확하게 쓸 수 있을 때. 반대로 기존 코드를 수정하거나, 요구사항이 계속 바뀌거나, 시각적 UI 검증이 핵심인 작업엔 맞지 않는다.

## 2. BMAD: AI 팀을 꾸린다는 아이디어

### 왜 이게 탄생했나

BMAD는 다른 문제의식에서 출발했다. 개발자 Brian Madison이 겪었던 건 아마 이런 상황이었을 것이다: AI에게 "이 앱을 만들어줘"라고 하면 그럴싸한 코드가 나오는데, 막상 쓰다 보면 PM적 관점에서의 기획이 빠져있거나, 아키텍처 결정이 성급하거나, UX가 엉망이다. AI가 단일 관점에서만 생각하기 때문이다.

BMAD의 해법은 **AI를 전문가 팀으로 나누는 것**이었다. 한 AI가 모든 걸 결정하게 하지 말고, 역할이 다른 여러 에이전트가 서로 견제하면서 결정을 내리게 한다.

### 실제로 설치해보니

```bash
npx bmad-method install --yes --modules bmm --tools claude-code --communication-language Korean
```

설치하고 나면 `.claude/skills/` 디렉토리에 43개의 스킬 파일이 생긴다. 하나하나 열어보면 각 에이전트의 역할이 정밀하게 정의되어 있다.

7개 에이전트가 역할을 나눠 갖는다:

| 이름 | 역할 | 담당 단계 |
|------|------|----------|
| Mary (Analyst) | 시장/도메인 리서치, 제품 브리프 | Phase 1 |
| John (PM) | PRD 작성, 스토리 관리 | Phase 2 |
| Winston (Architect) | 시스템 아키텍처, 기술 결정 | Phase 3 |
| Amelia (Developer) | 구현, 코드 리뷰, QA | Phase 4 |
| Sally (UX Designer) | UX 아티팩트 | Phase 2 옵션 |
| Paige (Tech Writer) | 기술 문서 | 필요 시 |
| BMad Master | 전체 오케스트레이션 | 전 단계 |

특이한 기능이 두 개 있다. **Party Mode**는 한 대화에서 여러 에이전트가 동시에 참여해 실제 팀 미팅처럼 토론한다. **적대적 리뷰(Adversarial Review)** 는 리뷰어에게 "문제 없다"는 답을 허용하지 않는다. 반드시 문제를 찾아야 한다는 규칙이다.

### 왜 43,000 스타인가

BMAD의 철학이 인상적이다:

> "Traditional AI tools do the thinking for you, producing average results. BMad agents guide you through a structured process to bring out **your best thinking**."

이게 핵심이다. AI를 "대신 생각해주는 기계"로 쓰면 평균적인 결과밖에 못 받는다. BMAD는 AI를 "최고의 사고를 이끌어내는 전문 협업자"로 위치시킨다. 인간이 주체이고, AI가 각 영역의 전문가 역할을 하면서 인간의 판단을 끌어낸다는 구조다. 이 메시지가 많은 개발자에게 공감을 얻었을 것이다.

### 언제 쓸까

혼자 일하는데 PM, 아키텍트, 개발자 역할이 동시에 필요할 때. 새 서비스를 처음부터 기획하고 만들 때 특히 유용하다. 단순한 기능 추가나 버그 수정에는 오버스펙이다.

## 3. Superpowers: 실패에서 태어난 규칙들

### 왜 이게 탄생했나

Superpowers는 조금 다른 관점에서 접근한다. AI 코딩 에이전트를 쓰다 보면 반복적으로 만나는 좌절이 있다: AI가 "완료했습니다"라고 말하는데 실제로는 안 된다. "아마 작동할 겁니다"라고 하는데 실제로 실행해보면 에러가 난다. 테스트도 없이 코드를 짜놓고 "나중에 추가하겠습니다"라고 한다.

Jesse Vincent는 이 패턴들을 **24건의 실패 사례**로 기록했고, 그 실패들에서 규칙을 뽑아냈다. Superpowers는 그 규칙들의 집합이다.

### 철학의 핵심

4가지 원칙 중 가장 인상적인 건 **증거 기반 검증**이다:

> "아마 될 거야"는 허용되지 않는다. 실행하고 확인해라.

AI가 "should work", "probably", "seems to" 같은 표현을 쓰면 그것 자체가 경고 신호다. 완료를 선언하기 전에 반드시 명령을 실행하고 출력을 확인해야 한다는 규칙이다.

TDD 원칙도 타협 없이 강제된다. 테스트 없이 작성된 코드는 삭제하고 TDD로 다시 짜야 한다. "테스트 후 작성(test-after)"은 TDD가 아니라고 명시적으로 규정한다.

### 14개 스킬의 구조

스킬들이 개발 라이프사이클 전체를 커버한다:

```
brainstorming → writing-plans → using-git-worktrees
→ executing-plans / subagent-driven-development
→ requesting-code-review → finishing-a-development-branch
```

흥미로운 건 각 스킬에 AI가 흔히 쓰는 "자기 합리화 패턴"을 차단하는 장치가 내장되어 있다는 점이다. 예를 들어 TDD 스킬에는 "이건 간단해서 테스트 불필요", "수동 테스트로 충분" 같은 변명들이 명시적으로 금지 패턴으로 등록되어 있다.

메타적으로 흥미로운 건, 스킬 자체에도 TDD를 적용한다는 것이다. 스킬 문서가 실제로 AI 행동을 바꾸는지 테스트하고 검증한다. 프로세스 문서의 품질까지 검증하는 접근이다.

### 직접 써보니

이 보고서 자체가 Superpowers가 설치된 환경에서 Claude Code와 함께 작성됐다. 브레인스토밍 → 플랜 작성 → 실행이라는 흐름을 실제로 경험해보니, 가장 체감되는 차이는 "AI가 멋대로 진행하는 것을 막는 구조"가 있다는 점이었다. 단계마다 확인을 요구하고, 증거 없이 완료를 선언하지 못하게 한다.

### 언제 쓸까

품질이 최우선인 프로젝트. TDD를 강제하고 싶을 때. AI 에이전트의 독단적 행동을 통제하고 싶을 때. 멀티 플랫폼(Claude, Cursor, Gemini 등)을 오가며 쓰는 경우에도 유용하다.

## 4. MOAI ADK: 한국에서 나온 하네스 엔지니어링

### 왜 이게 탄생했나

MOAI ADK는 네 개 중 유일하게 한국 팀(modu-ai)이 만들었다. 그리고 가장 도발적인 슬로건을 갖고 있다:

> "바이브 코딩의 목적은 빠른 생산성이 아니라 **코드 품질**이다."

이 한 줄이 MOAI ADK의 탄생 배경을 설명한다. AI 코딩 붐이 일면서 많은 개발자들이 "빠르게 뭔가 만들 수 있다"는 것에 집중했는데, modu-ai는 반대 방향으로 갔다. 빠름이 아니라 제대로를 선택한 것이다.

이들이 제안하는 개념이 **하네스 엔지니어링(Harness Engineering)** 이다. 인간 개발자의 역할이 "코드를 직접 작성하는 것"에서 "AI 에이전트가 올바르게 작동할 수 있는 환경(하네스)을 설계하는 것"으로 전환된다는 패러다임이다. 인간이 하는 일: SPEC 문서 작성, 품질 게이트 설정, 피드백 루프 구성.

### 규모가 다르다

설치하고 나면 규모에 압도된다. 24개 전문 에이전트, 52개 스킬. 비교하면: Ralph는 파일 수십 개, Superpowers는 14개 스킬, BMAD는 43개 스킬인데, MOAI ADK는 둘을 합친 것보다 크다.

에이전트들은 역할에 따라 분류된다:

- **Manager 8개**: spec, ddd, tdd, docs, quality, project, strategy, git
- **Expert 8개**: backend, frontend, security, devops, performance, debug, testing, refactoring
- **Builder 3개**: 새 에이전트/스킬/플러그인을 만드는 메타 에이전트
- **Evaluator 1개**: 독립적으로 품질을 회의적 시각으로 평가
- **Agency 6개**: 자기 진화형 크리에이티브 파이프라인

흥미로운 건 Builder 카테고리다. MOAI ADK가 스스로를 확장할 수 있는 에이전트가 내장되어 있다.

### 토큰 효율을 설계에 녹였다

52개 스킬을 전부 로드하면 토큰이 폭발한다. MOAI ADK는 이를 **프로그레시브 디스클로저(Progressive Disclosure)** 로 해결한다:

- **L1 메타데이터**: ~100 토큰, 항상 로드
- **L2 본문**: ~5K 토큰, 트리거 매칭 시에만 로드
- **L3 번들**: 필요할 때만 온디맨드

초기 토큰 소비가 67% 절감된다고 한다. 방법론의 철학(품질 중심)이 기술 설계에까지 반영된 것이다.

### SPEC 기반 워크플로우

모든 작업은 SPEC 문서에서 시작한다:

```bash
/moai plan "사용자 인증 기능 추가"  # SPEC 문서 생성
/moai run SPEC-AUTH-001              # TDD 또는 DDD로 자동 선택해 구현
/moai sync SPEC-AUTH-001             # 문서 동기화 및 PR 생성
```

프로젝트에 테스트 커버리지가 10% 이상이면 TDD 방식을, 10% 미만이면 DDD(기존 코드 보존 중심) 방식을 자동으로 선택한다.

### 설치 경험

PowerShell 7이 필요해서 먼저 `winget install Microsoft.PowerShell`로 설치했다. 그 후 한 줄로 완료됐다:

```powershell
irm https://raw.githubusercontent.com/modu-ai/moai-adk/main/install.ps1 | iex
```

Go로 만들어진 단일 바이너리라 설치 자체는 깔끔하다. 외부 의존성 없이 ~5ms 안에 실행된다.

### 언제 쓸까

Claude Code를 주 도구로 쓰는 경우, 중대형 프로젝트에서 품질 보증을 체계적으로 하고 싶을 때. 단, 학습 곡선이 가파르다. 24개 에이전트, 52개 스킬, SPEC 워크플로우까지 전부 이해하려면 상당한 시간 투자가 필요하다.

## 5. 네 가지를 비교하고 나서 든 생각

### 각자가 해결하려는 문제가 다르다

네 방법론이 같은 영역을 다루면서도 완전히 다른 문제를 겨냥하고 있다:

| 방법론 | 핵심 문제의식 | 해법 |
|--------|------------|------|
| Ralph | "컨텍스트 한계 때문에 AI가 망가진다" | 매번 새로 시작, 상태는 파일로 |
| BMAD | "단일 AI는 편협하게 생각한다" | 전문가 팀으로 분리 |
| Superpowers | "AI가 완료했다고 해도 실제로는 안 된다" | 규칙으로 행동을 강제 |
| MOAI ADK | "AI 코딩이 빠른 대신 품질이 무너진다" | 하네스 설계로 품질 보장 |

### AI와 인간의 역할을 어떻게 나누는가

- **Ralph**: 인간은 기획(PRD 작성)만, 실행은 AI가 자율적으로
- **BMAD**: 인간이 최종 의사결정, AI는 인간의 사고를 이끌어내는 역할
- **Superpowers**: 인간이 감독, AI는 규칙 안에서 실행 (자율성을 제한)
- **MOAI ADK**: 인간은 환경 설계, AI는 그 환경 안에서 실행

Ralph는 AI 자율성을 최대로, Superpowers는 가장 강하게 통제한다. BMAD와 MOAI는 그 중간 어딘가에 있는데, BMAD는 인간의 사고를 중심에 두고, MOAI는 환경 설계를 중심에 둔다.

### 공통적으로 보이는 흐름

세 가지가 눈에 띈다.

**첫째, SPEC/PRD 문서가 중심이 된다.** 네 방법론 모두 "먼저 무엇을 만들지 명확히 정의하고 시작"을 강조한다. 즉흥적 개발을 허용하지 않는다.

**둘째, 품질 게이트가 다층화된다.** typecheck, lint, test를 자동으로 통과해야 다음 단계로 넘어갈 수 있게 강제한다.

**셋째, AI와 인간의 역할을 명확히 분리한다.** 무엇이 인간의 판단 영역이고 무엇이 AI의 실행 영역인지를 구조적으로 정의한다.

---

## 6. 내 프로젝트에 어떻게 쓸 것인가

솔직히 이 네 가지를 다 동시에 쓰는 건 현실적이지 않다. 각각을 언제 어떻게 꺼낼지 생각해봤다.

**Ralph는 새 프로젝트의 빠른 뼈대를 잡을 때.** 처음부터 만드는 프로젝트에서, PRD를 충분히 구체적으로 쓸 수 있다면, Ralph 루프를 한번 돌려서 초기 구조를 만드는 데 써볼 수 있다. 단, 요구사항이 바뀌거나 기존 코드와 엮이는 순간 손을 떼야 한다.

**BMAD는 서비스 기획 단계에서.** 아이디어는 있는데 기획이 막막할 때, PM-아키텍트-개발자 관점을 번갈아가며 검토해주는 BMAD를 써볼 것 같다. 혼자 작업하면서 다양한 관점을 놓치지 않으려 할 때 특히.

**Superpowers는 일상 개발의 기준으로.** 이미 전역으로 설치되어 있고, 이 보고서 자체도 Superpowers 환경에서 작성됐다. TDD와 증거 기반 검증 원칙은 AI 도구를 쓸 때의 기본 습관으로 가져가려 한다.

**MOAI ADK는 품질이 중요한 프로젝트에서.** 학습 곡선이 높아서 모든 프로젝트에 쓰기는 어렵다. 하지만 코드 품질을 체계적으로 관리해야 하는 경우 SPEC 기반 워크플로우와 다층적 품질 게이트가 유용할 것 같다.

개인적으로 가장 오래 기억에 남을 것 같은 건 MOAI ADK의 "하네스 엔지니어링" 개념이다. AI에게 코드를 짜달라고 하는 것에서, AI가 잘 작동하는 환경을 설계하는 것으로 역할이 바뀐다는 발상. 아직 완전히 소화하진 못했지만, AI 도구를 쓰는 방식 자체를 다시 생각하게 만들었다.
