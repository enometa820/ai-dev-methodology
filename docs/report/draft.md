## 들어가며

이번 과제로 네 가지 AI 개발 방법론을 조사하게 됐다. 솔직히 처음엔 "방법론이 뭔 상관이야, 그냥 AI한테 시키면 되는 거 아닌가?" 싶었다. 그런데 직접 GitHub에 들어가서 각 프로젝트를 살펴보고, 실제로 설치해서 써보면서 생각이 달라졌다.

네 방법론이 해결하려는 문제가 전부 다르다. 그리고 "AI와 함께 개발한다"는 같은 방향을 향하면서도 접근 방식이 완전히 다르다. 조사하면서 각자의 탄생 배경과 철학을 이해하는 게 가장 재미있었다.

단순히 문서를 읽는 것에서 그치지 않고 네 가지 모두 실제로 설치해서 확인했다. 이 글은 그 과정에서 발견한 것들과, 앞으로 내 작업에 어떻게 쓸 수 있을지에 대한 기록이다.

## 1. Ralph: 될 때까지 무한 반복

### 이게 뭔가

Ralph의 작동 원리를 가장 정직하게 표현하면 이렇다:

> "기획서를 주면 AI가 혼자 검증 루프를 돌린다. 통과할 때까지."

원작자 Geoffrey Huntley가 공개한 코드의 핵심은 단 한 줄이다:

```
기획서를 읽고 → 만들고 → 확인하고 → 안 되면 다시 → 될 때까지 반복
```

사람이 중간에 개입하지 않는다. AI가 스스로 루프를 돌면서 모든 기능을 하나씩 완성한다. 무식하게 단순하지만, 이게 핵심이다.

### 왜 이 방식인가

AI와 긴 대화를 이어가다 보면 AI가 점점 이상해진다. 앞에서 한 말을 잊어버리고, 앞뒤가 맞지 않게 되고, 오류가 쌓인다. AI가 한 번에 기억할 수 있는 대화 분량에 구조적 한계가 있기 때문이다.

Ralph의 해법은 이 한계를 정면으로 피한다. 하나의 긴 대화 대신, 매 반복마다 AI를 완전히 새로 시작한다. 기억은 파일로 남긴다. `progress.txt`라는 파일에 "이번 반복에서 배운 것"을 계속 쌓아두고, 다음 AI가 그걸 이어받는 구조다.

덕분에 긴 대화가 누적되어 AI가 망가지는 현상을 구조적으로 차단한다.

### 실제로 어떻게 돌아가나

기획서를 작성한다. "로그인 기능 만들기", "상품 목록 보여주기" 같은 할 일들을 목록으로 정리해두는 것이다. Ralph가 실행되면:

```
1. 기획서에서 아직 안 된 일 중 우선순위 높은 것 하나 선택
2. AI가 그것만 집중해서 구현
3. 품질 자동 검증 실행
4. 통과하면 "완료" 표시 → 다음 일로
   통과 못 하면 다시 시도 (루프 재시작)
5. 전부 통과될 때까지 멈추지 않음
6. 모든 일 완료 시 자동 종료
```

`progress.txt`에 쌓인 학습 기록이 새 루프에 전달되어, 매번 새로 시작하면서도 이전 경험은 유지된다.

### 탄생 배경과 반응

원작자가 공개한 사례가 인상적이다. "$50,000 규모 MVP를 $297의 AI 비용으로 완성했다"고 했다. 과장이 있더라도 방향성은 분명하다 — AI를 "계속 지켜봐야 하는 도구"가 아니라 "혼자 돌아가는 공장"처럼 쓴다는 개념.

이 발상이 GitHub에서 14,000개 이상의 북마크를 받았다. 개발자들이 공감했다는 뜻이다.

### 한계와 적합한 상황

원작자 스스로 "기존에 만들어진 프로젝트에는 쓰지 않겠다"고 밝혔다. 이미 복잡하게 얽혀 있는 코드를 AI가 매번 새로 파악하기 어렵기 때문이다. Ralph는 완전히 새로운 것을 처음부터 만들 때, 무엇을 만들지 명확히 정해진 상황에서 빛을 발한다. 계획이 계속 바뀌거나 눈으로 확인해야 하는 시각적 작업이 중심인 경우엔 맞지 않는다.

## 2. BMAD: AI에게 팀을 구성해준다

### 이게 뭔가

BMAD는 AI 에이전트 하나에게 모든 걸 맡기는 대신, 역할이 다른 여러 AI가 팀을 이루어 함께 일하게 하는 프레임워크다. "AI팀 시뮬레이션"이라고 보면 된다.

설치하면 7명의 AI 전문가가 생긴다:

| 이름 | 역할 |
|------|------|
| Mary (분석가) | 브레인스토밍, 시장 조사, 아이디어 검토 |
| John (기획자) | 기능 명세서 작성, 우선순위 결정 |
| Winston (설계자) | 전체 구조 설계 |
| Amelia (개발자) | 실제 구현, 검토 |
| Sally (UX 디자이너) | 사용자 경험 설계 |
| Paige (문서 담당) | 기술 문서 작성 |
| BMad Master | 전체 에이전트 조율 |

### 왜 이 방식인가

AI에게 "이 서비스 만들어줘"라고 하면 뭔가 만들어주긴 한다. 그런데 막상 쓰다 보면 허술하다. 기획자 눈에 보이는 문제, 설계자 눈에 보이는 문제, 사용자 입장의 문제가 각각 다른데, AI 하나는 보통 한 가지 관점에서만 생각하기 때문이다.

개발자 Brian Madison이 만든 BMAD의 핵심 철학:

> "기존 AI 도구들은 AI가 대신 생각해줘서 평균적인 결과를 낸다. BMAD는 당신 스스로 최고의 사고를 하도록 이끄는 전문가 협업자 역할을 한다."

이게 핵심이다. AI한테 "알아서 해줘"라고 하면 평균적인 결과밖에 못 받는다. BMAD는 역할이 다른 AI들이 내 생각을 각자의 전문 영역에서 끌어내고 검증하는 구조다.

### 어떻게 진행되나

BMAD는 프로젝트를 4단계로 나눈다:

```
1단계 분석: Mary가 시장 조사, 아이디어 정리
2단계 기획: John + Sally가 기능 명세서, UX 설계
3단계 설계: Winston이 전체 구조 설계
4단계 구현: Amelia가 실제 제작, 반복 검토
```

단순한 작업이면 1~3단계를 합쳐서 빠르게 처리할 수도 있다. 복잡한 프로젝트일수록 각 단계를 제대로 밟는 게 중요하다.

흥미로운 기능이 두 가지 있다.

**Party Mode**: 한 대화 안에서 여러 AI가 동시에 참여해 실제 팀 회의처럼 토론한다. PM이 의견을 내면 설계자가 반박하고, 개발자가 현실적인 우려를 제기하는 식이다.

**적대적 리뷰**: 리뷰하는 AI가 "문제없습니다"라는 답을 절대 낼 수 없다. 반드시 문제점을 찾아야만 하는 규칙이다. AI가 습관적으로 긍정적인 답변을 내놓는 걸 구조적으로 막는 장치다.

또 하나 인상적인 게 `project-context.md`라는 파일이다. 7명의 AI 모두가 참고하는 일종의 "헌법 문서"로, 프로젝트의 기술 선택, 규칙, 방향을 여기에 명시해두면 모든 AI가 일관된 방향으로 움직인다.

### 얼마나 쓰이나

GitHub 북마크 43,000개 이상. 조사한 네 방법론 중 압도적 1위다. "혼자 일하지만 팀처럼 생각하고 싶다"는 수요가 얼마나 큰지 보여준다.

### 언제 쓸까

새 서비스를 처음부터 체계적으로 기획하고 만들 때. 혼자 일하면서 기획자, 설계자, 개발자 관점을 동시에 챙겨야 할 때 특히 유용하다. 간단한 작업 하나를 처리하는 데는 과하다.

## 3. Superpowers: 실패 24번에서 나온 규칙들

### 이게 뭔가

Superpowers는 AI 에이전트에게 14개의 행동 규칙(스킬)을 주입하는 프레임워크다. 각 규칙은 AI가 작업 중 흔히 저지르는 실수를 막기 위해 설계됐다.

특이한 점은 탄생 배경이다. 개발자 Jesse Vincent가 AI 도구를 쓰면서 겪은 실패 사례 24가지를 직접 기록했고, 그 실패들에서 규칙을 뽑아냈다. Superpowers는 실험실에서 만든 이론이 아니라 실패에서 나온 교훈의 집합이다.

### 왜 이 방식인가

AI 도구를 쓰다 보면 반복적으로 만나는 좌절이 있다.

- "완료했습니다"라고 하는데 실제로는 안 된다
- "아마 잘 작동할 겁니다"라고 하는데 막상 써보면 오류가 난다
- 확인도 안 하고 "됩니다"라고 선언한다

Jesse Vincent는 이 패턴들을 24가지로 정리했고, 각 패턴에 대응하는 규칙을 만들었다.

가장 핵심적인 원칙:

> "아마 될 거야는 허용되지 않는다. 직접 실행해보고 결과를 확인해라."

AI가 "아마도", "될 것 같습니다", "작동할 것으로 보입니다" 같은 표현을 쓰면, 그것 자체가 경고 신호다. 완료를 선언하기 전에 반드시 실제로 실행해보고 눈으로 결과를 확인해야 한다.

또 하나 인상적인 원칙이 있다. **기능을 만들기 전에 먼저 "어떻게 되면 성공인지"를 정의하는 것을 의무화한다.** 성공 기준 없이 만들어진 결과물은 지우고 다시 시작해야 한다.

### 14개 규칙의 구조

규칙들이 작업의 전체 흐름을 빈틈없이 커버한다:

```
아이디어 정리 → 계획 작성 → 작업 공간 준비
→ 실행 → 결과 검토 → 마무리
```

흥미로운 건 각 규칙에 AI가 흔히 쓰는 "핑계"를 차단하는 장치가 내장되어 있다는 점이다. "이건 단순해서 확인 안 해도 돼", "나중에 고치면 돼" 같은 자기 합리화가 명시적으로 금지되어 있다.

한 가지 더 — Superpowers는 Claude Code 외에 Cursor, Gemini, GitHub Copilot 등 여러 AI 도구에서 동일하게 사용할 수 있다.

### 직접 써보니

이 보고서 자체가 Superpowers가 설치된 환경에서 AI 도구(Claude Code)와 함께 작성됐다. 체감되는 가장 큰 차이는 "AI가 멋대로 진행하는 것을 막는 구조"였다. 단계마다 확인을 요구하고, 실제로 해보지 않고는 완료라고 말하지 못하게 한다.

### 언제 쓸까

품질이 최우선인 작업. "AI가 됐다고 했는데 나중에 보니 안 됐더라"는 경험을 줄이고 싶을 때. 이미 설치되어 있으므로 기본값으로 사용하고 있다.

## 4. MOAI ADK: 한국에서 나온 환경 설계 방법론

### 이게 뭔가

MOAI ADK는 네 개 중 유일하게 한국 팀(modu-ai)이 만들었다. GitHub 공개 문서와 공식 사이트(adk.mo.ai.kr)가 한국어로 제공된다.

가장 도발적인 슬로건을 갖고 있다:

> "AI 코딩의 목적은 빠른 생산성이 아니라 **코드 품질**이다."

이 한 줄이 MOAI ADK의 탄생 배경을 설명한다. AI 코딩 열풍이 불면서 많은 사람들이 "AI로 빠르게 뭔가 만들 수 있다"는 것에 집중했는데, modu-ai는 반대 방향을 선택했다. 빠름이 아니라 제대로를 목표로 했다.

### 핵심 개념: 환경 설계

이들이 제안하는 개념이 **환경 설계(Harness Engineering)** 다.

기존 방식: 사람이 "이거 만들어줘" → AI가 실행  
환경 설계 방식: 사람이 AI가 잘 작동할 조건을 설계 → AI가 그 안에서 실행

사람이 해야 할 일이 "직접 만드는 것"에서 "무엇을 만들지 상세히 정의하기, 품질 기준 세우기, 피드백 구조 만들기"로 바뀐다.

처음엔 이게 무슨 말인지 잘 몰랐는데, 써보면서 조금씩 이해됐다. 내가 상세하게 기획서를 쓸수록 AI의 결과물이 달라진다는 것.

### 규모

설치하면 26명의 전문 AI와 52개의 작업 규칙이 준비된다.

- **관리자 8명**: 기획서 작성, 전략 수립, 문서화, 품질 관리 등
- **전문가 8명**: 각 기술 영역 담당 (웹, 보안, 성능, 오류 수정 등)
- **제작자 3명**: 새로운 AI나 규칙을 만드는 메타 역할
- **평가자 1명**: 독립적이고 비판적인 시각으로 품질 평가
- **에이전시 6명**: 창의적 작업 전담

비교하면 Superpowers가 14개 규칙, BMAD가 7명의 AI인데, MOAI ADK는 이들과 맞먹는 규모다. 특히 "제작자" 카테고리가 흥미롭다 — MOAI ADK 자체를 확장할 수 있는 AI가 내장되어 있다는 의미다.

### 품질 관리 구조

MOAI ADK에서 인상적이었던 부분 중 하나는 **TRUST 5** 프레임워크다. 작업 복잡도에 따라 검증 수준을 3단계로 나눈다:

- 단순 변경 → 빠른 검증
- 일반 작업 → 기본 품질 체크
- 복잡한 기획 → 전체 품질 게이트 통과 필수

그리고 **MX 태그**라는 시스템이 있다. 코드 안에 작은 메모를 남기는 방식인데, "이 부분은 변경 불가", "주의 필요", "나중에 처리할 것" 같은 정보를 AI가 읽을 수 있는 형태로 코드에 직접 붙인다. AI가 작업할 때 이 메모를 참고해서 실수를 줄이는 구조다.

### 비용 절감 설계

AI에게 52개 규칙을 한 번에 전부 읽히면 비용이 폭발한다. MOAI ADK는 이를 단계적 공개 방식으로 해결한다:

- 항상 준비된 기본 정보 (매우 적은 양)
- 필요한 상황이 되면 그때 불러오는 내용
- 특별히 필요할 때만 추가로 불러오는 상세 내용

이 방식으로 초기 비용이 67% 절감된다고 한다. "품질 중심"이라는 철학이 비용 설계에까지 반영된 것이다.

### 어떻게 사용하나

모든 작업은 상세 기획서에서 시작한다:

```
"사용자 로그인 기능 추가"라고 지시
→ AI가 상세 기획서 자동 생성

해당 기획서 번호로 실행 지시
→ AI가 프로젝트 상태에 맞는 방식으로 자동 선택해 구현

동기화 지시
→ 문서 정리 및 검토 요청 자동 생성
```

프로젝트 품질 수준에 따라 AI가 작업 방식을 자동으로 선택한다. 새 프로젝트라면 성공 기준을 먼저 정하고 만드는 방식으로, 기존 프로젝트라면 현재 코드를 최대한 보존하면서 개선하는 방식으로.

### 설치 경험

Windows 기본 환경에는 구버전 도구가 있어서 먼저 PowerShell 7을 추가 설치해야 했다. 그 과정까지 포함해서 결국 두 줄의 명령어로 끝났다. Go라는 언어로 만들어진 단독 실행 파일이라 추가로 뭔가를 더 설치할 필요가 없었다.

### 언제 쓸까

Claude Code를 주 도구로 쓰는 경우, 품질 관리를 체계적으로 하고 싶은 규모 있는 프로젝트에서. 단, 배우는 데 시간이 필요하다. 26명의 AI, 52개 규칙, 상세 기획서 기반 작업 방식까지 전부 익히려면 상당한 투자가 필요하다. GitHub 북마크가 900개로 다른 방법론에 비해 적은데, 그만큼 아직 초기 단계라는 의미도 있다.

## 5. 네 가지를 비교하고 나서 든 생각

### 각자가 해결하려는 문제가 다르다

조사하면서 가장 흥미로웠던 건, 네 방법론이 완전히 다른 문제를 겨냥하고 있다는 점이었다:

| 방법론 | 핵심 문제의식 | 해법 |
|--------|------------|------|
| Ralph | "대화가 길어지면 AI가 망가진다" | 검증 통과할 때까지 루프 반복 |
| BMAD | "AI 하나는 편협하게 생각한다" | 역할 다른 AI 팀으로 분리 |
| Superpowers | "AI가 됐다고 해도 실제로는 안 된다" | 규칙으로 검증을 강제 |
| MOAI ADK | "빠름이 우선되면 품질이 무너진다" | 환경 설계로 품질 보장 |

### AI와 나의 역할을 어떻게 나누는가

- **Ralph**: 내가 기획서를 쓰면, 실행은 AI가 알아서 루프 돌림
- **BMAD**: AI 팀이 내 생각을 끌어내고 검증, 최종 결정은 내가
- **Superpowers**: 내가 감독, AI는 정해진 규칙 안에서만 실행
- **MOAI ADK**: 내가 환경 설계, AI는 그 환경 안에서 실행

Ralph는 AI 자율성을 최대로 주고, Superpowers는 가장 강하게 통제한다. BMAD와 MOAI ADK는 그 중간이다.

### 공통으로 보이는 흐름

세 가지가 눈에 띈다.

**첫째, 즉흥적으로 시작하지 않는다.** 네 방법론 모두 "AI한테 일단 해봐"를 경계한다. Ralph는 기획서를, BMAD는 명세서를, Superpowers는 성공 기준을, MOAI ADK는 상세 기획서를 먼저 요구한다.

**둘째, 자동 검증이 필수다.** 작업이 끝나면 정해진 기준들을 자동으로 통과했는지 확인한다. "됐을 것 같다"는 통하지 않는다.

**셋째, AI와 인간의 역할을 명확히 나눈다.** 무엇이 내가 결정해야 하는 영역이고 무엇이 AI가 실행하는 영역인지를 구조적으로 정의한다.

## 6. 내 프로젝트에 어떻게 쓸 것인가

솔직히 네 가지를 동시에 다 쓰는 건 현실적이지 않다. 각각을 언제 꺼낼지 생각해봤다.

**Ralph는 완전히 새로운 것을 처음 만들 때.** 무엇을 만들지 명확히 정리되어 있고, 처음부터 새로 시작하는 프로젝트라면 Ralph를 돌려서 기본 뼈대를 빠르게 만드는 데 써볼 것 같다. 단, 기존 작업물과 섞이는 순간 손을 떼야 한다.

**BMAD는 기획 단계에서.** 아이디어는 있는데 어디서 어떻게 시작해야 할지 막막할 때, 7명의 AI 전문가 관점을 빌려서 빈틈을 찾는 데 활용해볼 것 같다. 혼자 작업하면서 다양한 시각을 놓치지 않으려 할 때 특히.

**Superpowers는 일상적인 작업 기준으로.** 이미 설치되어 있고, 이 보고서 자체도 Superpowers 환경에서 작성됐다. "AI의 말을 그냥 믿지 말고 직접 확인하라"는 원칙은 AI 도구를 쓸 때의 기본 습관으로 가져가려 한다.

**MOAI ADK는 품질이 정말 중요한 프로젝트에서.** 학습하는 데 시간이 필요해서 모든 작업에 쓰기는 어렵다. 하지만 제대로 만드는 게 중요한 규모 있는 프로젝트에서 체계적인 품질 관리 도구로 활용하고 싶다.

개인적으로 가장 오래 기억에 남을 것 같은 건 MOAI ADK의 "환경 설계" 개념이다. 지금까지는 AI에게 "이거 해줘"라고 시키는 사람이었는데, AI가 잘 작동할 수 있는 조건을 내가 설계하는 사람이 되어야 한다는 발상. 이번 조사를 통해 AI 도구를 대하는 방식 자체를 다시 생각하게 됐다.
