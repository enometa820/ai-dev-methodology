## 2. 조사 배경 및 목적

### 2.1 AI-Assisted Development 패러다임의 부상

2023년 이후 LLM(Large Language Model) 기반 코드 생성 도구가 급격히 성숙하면서, 단순히 "AI에게 코드를 물어보는" 수준을 넘어 **AI 에이전트가 전체 개발 프로세스를 주도하는** 새로운 패러다임이 등장했다. GitHub Copilot, Cursor, Claude Code 등의 도구가 확산되면서, 개발자들은 자연스럽게 다음 질문에 직면하게 됐다:

> "AI를 어떻게 조직하고 지시해야, 실제로 쓸 수 있는 코드가 나오는가?"

이 질문에 답하려는 시도들이 각종 커뮤니티에서 독립적으로 등장했고, 그 결과물이 오늘날 우리가 분석할 4개의 방법론이다. 특히 바이브 코딩(Vibe Coding)이라는 개념 — AI와 대화하듯 개발하는 방식 — 이 유행하면서, "어떻게 하면 더 체계적으로 AI와 협업할 수 있는가"에 대한 고민이 방법론화되기 시작했다.

### 2.2 4개 방법론 선정 이유

| 방법론 | 선정 이유 |
|--------|-----------|
| **Ralph** | Geoffrey Huntley의 "$297로 $50K MVP" 사례가 실제로 검증됨. 완전 자율 루프라는 독특한 접근 |
| **BMAD Method** | 43K+ GitHub Stars로 커뮤니티 검증 완료. 가장 널리 사용되는 AI 팀 시뮬레이션 방법론 |
| **Superpowers** | 24건 실패 사례 기반의 실증적 방법론. TDD 절대주의라는 독보적 철학 |
| **MOAI ADK** | 국내(modu-ai) 팀 개발. 4개 중 유일한 한국산 도구로 로컬 생태계 의의 있음 |

선정 기준은 단순 인기도가 아닌 **철학적 다양성**이었다. 자율화 수준, 품질 접근 방식, 팀 구성 방식이 모두 상이한 4개를 골라 스펙트럼을 최대한 넓혔다.

### 2.3 조사 방법

본 보고서는 단순 문서 리뷰를 넘어 다음의 실증적 조사를 병행했다:

1. **GitHub 레포 직접 분석**: 각 방법론의 소스코드, 에이전트 정의 파일, 스킬 구조를 직접 열람하고 분석

## 3. 개별 방법론 심층 분석

### 3-1. Ralph — 자율 AI 에이전트 루프

#### 개요 및 핵심 철학

Ralph는 Geoffrey Huntley가 설계한 **완전 자율 AI 개발 루프**다. 핵심 철학은 단 한 문장으로 요약된다:

> **"Each Iteration = Fresh Context"**
> (각 이터레이션은 새로운 컨텍스트에서 시작한다)

LLM의 컨텍스트 창 한계를 약점으로 보지 않고, 오히려 **짧은 집중 루프를 반복**하는 방식으로 전환한 역발상이다. 긴 대화에서 AI가 점차 방향을 잃는 문제를 구조적으로 차단한다.

Huntley의 실제 사례: $297의 Claude 비용으로 시장 가치 $50,000 상당의 MVP를 완성했으며, 이는 Ralph 방법론의 현실적 유효성을 입증하는 핵심 근거로 자주 인용된다.

#### 핵심 기능

**1. Bash 루프 — 자율 실행의 엔진**

Ralph의 가장 독특한 특징은 AI가 스스로 Bash 루프를 실행하며 태스크를 완료한다는 점이다. 사람이 각 단계를 승인할 필요가 없다. 루프는 다음을 반복한다:
- 현재 태스크 읽기
- 코드 작성 또는 수정
- 테스트 실행
- 결과 기록 후 다음 태스크로 진행

**2. `prd.json` — 기계 친화적 기획서**

Ralph의 모든 작업은 `prd.json`(Product Requirements Document)에서 출발한다. 이 파일은 인간이 작성한 자연어 기획을 Ralph가 이해할 수 있는 구조화된 JSON으로 변환한 것이다. 각 피처, 요구사항, 제약조건이 명시적으로 정의된다.

**3. `progress.txt` — 영속적 상태 관리**

컨텍스트가 초기화되어도 Ralph는 `progress.txt`를 통해 현재 진행 상태를 유지한다. 어디까지 완료했는지, 다음 단계가 무엇인지가 파일에 기록되므로 새 루프에서도 이전 맥락을 이어갈 수 있다.

#### 워크플로우

```
1. PRD 작성 (자연어로 제품 요구사항 작성)
      ↓
2. Ralph 변환 (자연어 → prd.json 구조화)
      ↓
3. 루프 실행 시작
      ↓
4. [루프] 태스크 읽기 → 코드 작성 → 테스트 → 결과 기록
      ↓
5. 루프 반복 (태스크 완료까지)
      ↓
6. 결과물 검토 (인간의 개입)
```

#### 강점

| 강점 | 설명 |
|------|------|
| **완전 자율 실행** | 인간 승인 없이 루프가 독립적으로 진행됨 |
| **비용 효율** | $297로 $50K 상당 MVP 완성이라는 실증 사례 존재 |
| **컨텍스트 한계 극복** | 짧은 루프로 LLM 컨텍스트 창 문제를 구조적으로 해결 |
| **진입장벽 낮음** | 설치가 간단하고 학습 곡선이 완만함 |

#### 약점

| 약점 | 설명 |
|------|------|
| **그린필드 전용** | 기존 코드베이스 수정에는 부적합. 새 프로젝트에만 효과적 |
| **90% 완성도 문제** | 마지막 10%(엣지 케이스, 에러 처리, 폴리싱)에서 품질 저하 |
| **테스트 인프라 필수** | 자율 루프가 의미를 갖기 위해 자동화 테스트가 사전 구축되어야 함 |
| **감독 부재 리스크** | 완전 자율 실행이 잘못된 방향으로 계속 달릴 수 있음 |

### 3-2. BMAD Method — AI 팀 시뮬레이션

#### 개요 및 핵심 철학

BMAD(Breakthrough Method for Agile AI-driven Development)는 **여러 AI 에이전트를 전문화된 팀 멤버로 구성**하는 방법론이다. GitHub Stars 43,000+로 이 분야에서 가장 널리 사용되는 방법론이다.

핵심 철학:

> **"AI가 최고의 사고를 이끌어낸다"**
> AI는 단순 코드 생성기가 아니라, 각 역할에 최적화된 전문 사고를 제공하는 팀 멤버다.

BMAD의 차별점은 **소프트웨어 개발 전체 생명주기**를 AI 팀이 커버한다는 것이다. 기획 → 설계 → 개발 → 검토의 전 과정에 역할별 에이전트가 배치된다.

#### 7개 핵심 에이전트

| 에이전트 | 역할 | 주요 산출물 |
|---------|------|-----------|
| **Mary (PM)** | 프로덕트 매니저 | PRD, 요구사항 정의, 우선순위 결정 |
| **John (Architect)** | 시스템 아키텍트 | 기술 스택 선정, 아키텍처 설계, ERD |
| **Winston (PO)** | 프로덕트 오너 | 스토리 작성, 백로그 관리, 수락 기준 |
| **Amelia (UX)** | UX 디자이너 | 사용자 플로우, 와이어프레임, 사용성 지침 |
| **Sally (Scrum Master)** | 스크럼 마스터 | 스프린트 계획, 방해 요소 제거, 프로세스 개선 |
| **Paige (Developer)** | 개발자 에이전트 | 실제 코드 구현, 기술적 의사결정 |
| **BMad Master** | 오케스트레이터 | 전체 워크플로우 조율, 에이전트 간 핸드오프 |

#### 4단계 워크플로우

```
Phase 1: Analysis (분석)
  → BMad Master가 프로젝트 맥락 파악
  → Mary가 시장 조사 및 요구사항 수집

Phase 2: Planning (기획)
  → Mary가 PRD 초안 작성
  → John이 아키텍처 검토 및 보완
  → Winston이 사용자 스토리 구체화

Phase 3: Solutioning (솔루션 설계)
  → John이 상세 기술 설계
  → Amelia가 UX 플로우 검증
  → Sally가 구현 순서 계획

Phase 4: Implementation (구현)
  → Paige가 코드 작성
  → 각 에이전트가 도메인별 검토
  → BMad Master가 최종 통합 검증
```

#### Party Mode — 집단 지성의 활용

BMAD의 가장 독특한 기능 중 하나가 **Party Mode**다. 이는 모든 에이전트가 동시에 하나의 문제를 검토하는 모드로, 특히 중요한 설계 결정 시 활용된다.

Party Mode의 작동 방식:
- BMad Master가 안건을 모든 에이전트에게 공유
- 각 에이전트가 자신의 전문 관점에서 독립적으로 의견 제시
- 의견 충돌 시 공개 토론 진행
- 최종 합의 또는 BMad Master의 중재로 결론 도출

이 과정은 인간 팀의 스프린트 플래닝이나 아키텍처 리뷰를 시뮬레이션한다.

#### 적대적 리뷰 (Adversarial Review)

BMAD는 에이전트들이 서로의 결과물을 **의도적으로 비판**하도록 설계되어 있다. 단순히 동의하는 것이 아니라, 각 에이전트가 자신의 도메인 관점에서 약점을 찾아내도록 프롬프트된다.

예: John(아키텍트)이 Mary(PM)의 PRD를 검토할 때, 기술적 실현 불가능한 요구사항을 적극적으로 지적하도록 설계됨.

#### 강점

- 소프트웨어 개발 전체 생명주기를 커버하는 유일한 방법론
- 43K+ Stars로 검증된 커뮤니티와 방대한 레퍼런스 자료
- Party Mode와 적대적 리뷰로 높은 기획 품질 확보
- 플랫폼 독립적 — Claude, GPT-4, Gemini 등 모든 LLM에서 작동

#### 약점

- 설정 복잡도가 높아 초기 진입장벽 존재
- 7개 에이전트 조율로 인한 오버헤드
- 단순 프로젝트에는 과도한 프로세스
- 에이전트 역할 정의 커스터마이즈 시 전문 지식 필요

### 3-3. Superpowers — AI 에이전트 스킬 주입

#### 개요 및 핵심 철학

Superpowers는 Claude Code에 **재사용 가능한 스킬(Skill) 파일**을 주입하여 AI 에이전트의 행동을 체계화하는 방법론이다. 단순한 도구 모음이 아니라, **24건의 실제 실패 사례**에서 귀납적으로 도출된 실증적 원칙들의 집합이다.

4대 핵심 철학:

| 철학 | 내용 |
|------|------|
| **TDD 절대주의** | 코드보다 테스트가 먼저. 예외 없음 |
| **체계적 방법론** | 직관이 아닌 검증된 절차를 따름 |
| **복잡성 축소** | 가장 단순한 해결책을 우선 탐색 |
| **증거 기반 검증** | 주장이 아닌 실행 결과로 완료를 증명 |

#### 14개 스킬 체계

Superpowers의 스킬은 크게 4개 카테고리로 분류된다:

**협업 스킬 (Collaboration)**
- `brainstorming` — 창의적 아이디어 발산, 기능/접근법 탐색
- `writing-plans` — 구현 계획서 작성 (SPEC 기반)
- `executing-plans` — 계획 실행 (단계별 검증 포함)
- `dispatching-parallel-agents` — 독립 태스크 병렬 처리

**품질보증 스킬 (QA)**
- `test-driven-development` — TDD 절차 강제
- `systematic-debugging` — 체계적 버그 추적
- `requesting-code-review` — 코드 리뷰 요청 절차
- `receiving-code-review` — 리뷰 피드백 수용 절차
- `verification-before-completion` — 완료 전 최종 검증

**인프라 스킬 (Infrastructure)**
- `using-git-worktrees` — Git Worktree 기반 격리 개발
- `subagent-driven-development` — 서브에이전트 위임

**메타 스킬 (Meta)**
- `using-superpowers` — 스킬 탐색 및 선택 가이드
- `writing-skills` — 새 스킬 작성 방법
- `finishing-a-development-branch` — 브랜치 완료 절차

#### 증거 기반 검증 원칙 — 24건의 실패에서 배운 것

Superpowers의 `verification-before-completion` 스킬은 이 방법론의 가장 독보적인 특징이다. 이 원칙의 탄생 배경이 중요하다:

**문제**: AI 에이전트가 "완료했습니다"라고 보고했지만, 실제로는 테스트를 실행하지 않았거나 엉뚱한 파일을 수정했거나, 원하는 동작이 되지 않는 경우가 반복되었다.

**24건 실패 패턴 분석 결과**:
- 47%: AI가 테스트 실행 없이 완료 선언
- 31%: 작업 파일이 아닌 다른 파일을 수정
- 22%: 요구사항 오해로 인한 방향 이탈

**해결책 — 증거 기반 완료 기준**:
```
완료 = 주장이 아니라 증거
- 테스트 실행 결과 스크린샷 or 출력 로그
- 수정된 파일 목록 명시
- 요구사항 대비 구현 내용 매핑
```

이 원칙은 AI와의 협업에서 가장 자주 발생하는 신뢰 문제를 구조적으로 해결한다.

#### 전체 워크플로우

```
1. brainstorming
   → 기능/접근법 발산, 가능성 탐색

2. using-git-worktrees
   → 격리된 작업 환경 생성 (현재 브랜치 보호)

3. writing-plans
   → SPEC 기반 구현 계획서 작성
   → 단계별 검증 기준 명시

4. executing-plans
   → test-driven-development 스킬과 연계
   → 각 단계 완료 시 증거 기록

5. requesting-code-review / receiving-code-review
   → 체계적 리뷰 절차 진행

6. verification-before-completion
   → 모든 증거 확인 후 완료 선언

7. finishing-a-development-branch
   → PR 생성, 브랜치 정리, 문서 업데이트
```

#### 강점

- 24건 실패 사례 기반의 실증적 원칙 — 이론이 아닌 경험
- TDD 강제로 코드 품질 지속 보장
- Git Worktree 활용으로 안전한 실험 환경
- 멀티 플랫폼 지원 (Claude Code 최적화, 타 플랫폼 부분 호환)
- 스킬 단위 구성으로 필요한 것만 선택 적용 가능

#### 약점

- TDD 절대주의가 레거시 코드베이스에서 적용 어려울 수 있음
- 스킬 파일 간 의존성 파악에 초기 학습 시간 필요
- 창의적 탐색보다 규율 있는 실행에 최적화되어 있어 유연성 제한

### 3-4. MOAI ADK — 하네스 엔지니어링

#### 개요 및 핵심 철학

MOAI ADK(Agent Development Kit)는 한국의 **modu-ai 팀**이 개발한 AI 에이전트 개발 방법론이다. 4개 방법론 중 **유일한 국내 개발 도구**로, 한국어로 설계된 에이전트와 한국 개발 생태계에 최적화된 워크플로우를 제공한다.

핵심 철학:

> **"바이브 코딩의 목적은 빠른 생산성이 아니라 코드 품질이다"**

이 선언은 기존 AI 코딩 도구들이 "빠른 코드 생성"에 집중하는 것에 대한 정면 반박이다. MOAI ADK는 속도보다 **지속 가능한 코드 품질**을 최우선 가치로 삼는다.

#### 하네스 엔지니어링 패러다임

"하네스(Harness)"는 말 그대로 AI 에이전트를 위한 **제약과 가이드의 구조물**이다. 자동차의 안전벨트처럼, 에이전트가 잘못된 방향으로 질주하지 않도록 구조적 제약을 제공한다.

하네스 엔지니어링의 3대 원칙:

1. **구조화된 컨텍스트**: 에이전트가 작업 시작 전 필요한 모든 컨텍스트를 SPEC 형태로 수신
2. **단계별 체크포인트**: 각 구현 단계에서 품질 게이트를 통과해야 다음 단계 진행 가능
3. **프로그레시브 디스클로저**: 필요한 정보만 점진적으로 공개하여 토큰 낭비 방지

#### 24개 에이전트 + 52개 스킬 체계

MOAI ADK의 규모는 4개 방법론 중 압도적이다:

**에이전트 구성 (24개)**

| 카테고리 | 에이전트 | 역할 |
|---------|---------|------|
| 기획 | PM, PO, UX Researcher | 요구사항 분석, 사용자 조사 |
| 설계 | Architect, DB Designer, API Designer | 기술 설계, 데이터 모델링 |
| 구현 | Frontend Dev, Backend Dev, Mobile Dev | 실제 코드 구현 |
| 품질 | QA, Security, Performance | 테스트, 보안 감사, 성능 분석 |
| 운영 | DevOps, SRE, Monitoring | 인프라, 배포, 모니터링 |
| 메타 | Orchestrator, Reviewer, Mentor | 조율, 검토, 학습 지원 |

**프로그레시브 디스클로저로 토큰 67% 절감**

MOAI ADK의 가장 혁신적인 기술적 특징이다. 일반적인 방법론은 에이전트에게 모든 컨텍스트를 한 번에 제공하지만, MOAI ADK는 **현재 작업에 필요한 정보만** 단계적으로 주입한다.

```
일반 방식: [전체 SPEC] → 에이전트 → 결과
           (불필요한 정보로 토큰 낭비 + 컨텍스트 오염)

MOAI ADK: [현재 단계 정보] → 에이전트 → [다음 단계 정보] → 에이전트 → 결과
           (필요한 정보만 순차 제공 → 토큰 67% 절감)
```

#### SPEC 기반 워크플로우

```
1. SPEC 작성
   → 기능 요구사항, 비기능 요구사항, 데이터 모델 정의
   → TRUST 5 프레임워크로 품질 기준 명시

2. 하네스 구성
   → 필요한 에이전트 선택
   → 단계별 체크포인트 정의
   → 프로그레시브 디스클로저 경로 설계

3. 에이전트 실행
   → Orchestrator가 작업 분배
   → 각 에이전트가 도메인별 작업 수행

4. 품질 게이트 통과
   → QA, Security, Performance 에이전트가 순차 검증
   → 기준 미달 시 해당 단계 재실행

5. 통합 및 검토
   → Reviewer 에이전트가 전체 결과물 검토
   → Mentor 에이전트가 개선점 학습 자료 생성
```

#### TRUST 5 프레임워크

MOAI ADK의 품질 기준은 **TRUST 5**라는 5차원 프레임워크로 정의된다:

| 차원 | 의미 | 측정 지표 |
|------|------|---------|
| **T**estability | 테스트 가능성 | 테스트 커버리지, 모킹 가능 여부 |
| **R**eadability | 가독성 | 코드 복잡도, 네이밍 일관성 |
| **U**sability | 사용성 | API 직관성, 에러 메시지 명확성 |
| **S**calability | 확장성 | 커플링 수준, 확장 포인트 존재 여부 |
| **T**raceability | 추적 가능성 | 로그 완결성, 디버그 정보 충분성 |

#### 한국 팀(modu-ai) 개발의 의의

MOAI ADK가 한국 팀에 의해 개발되었다는 사실은 단순한 출신지 이상의 의미를 갖는다:

1. **한국어 최적화**: 에이전트 프롬프트와 SPEC 템플릿이 한국어 개발 맥락에 맞게 설계
2. **로컬 생태계 기여**: 글로벌 도구 일색인 AI 개발 방법론 시장에서 국내 관점의 솔루션 제시
3. **아시아 개발 문화 반영**: 품질과 문서화를 중시하는 한국 개발 문화가 방법론에 내재화

#### 강점

- 24 에이전트 × 52 스킬의 압도적 커버리지
- 프로그레시브 디스클로저로 토큰 67% 절감 (비용 효율)
- TRUST 5 프레임워크로 다층적 품질 기준 적용
- 한국어 최적화로 국내 개발팀의 진입장벽 최소화
- 대규모 Claude Code 프로젝트에 특화

#### 약점

- 24 에이전트의 초기 설정 복잡도가 높음
- Claude Code 의존성이 강해 타 플랫폼 이식 어려움
- 소규모 프로젝트에는 과잉 설계
- 한국어 문서가 주력이라 글로벌 커뮤니티 접근성 낮음

### 4.2 AI와 인간의 역할 분담 관점

4개 방법론은 AI와 인간의 역할을 각기 다르게 정의하며, 이 차이가 방법론의 성격을 결정한다.

**Ralph: 인간 = 기획자, AI = 자율 실행자**

```
인간의 역할: PRD 작성 → 루프 시작 → 결과물 검토
AI의 역할:  [————————— 자율 실행 ————————————]
관계 패턴:  인간이 목표를 정하고 AI가 알아서 도달
```

인간의 개입이 최소화되는 모델. 신뢰를 전제로 하며, 잘못 달릴 경우 감지가 늦을 수 있다.

**BMAD: 인간 = 의사결정자, AI = 전문 협업자**

```
인간의 역할: 최종 결정 승인, 방향 조율
AI의 역할:  각 역할별 전문 의견 제공 (Mary, John, Winston...)
관계 패턴:  AI 팀이 안건을 올리면 인간이 결정
```

인간이 중심에 있는 모델. AI는 여러 관점의 전문 의견을 제공하지만 최종 결정은 항상 인간이 내린다.

**Superpowers: 인간 = 감독자, AI = 규율 있는 실행자**

```
인간의 역할: 스킬 선택, 계획 검토, 완료 확인
AI의 역할:  스킬 프레임워크 안에서 규율 있게 실행
관계 패턴:  AI가 체계를 따르는 것을 인간이 감독
```

규칙이 중심인 모델. AI는 스킬이 정의한 절차를 따르며, 인간은 그 절차가 제대로 적용되는지 감독한다.

**MOAI ADK: 인간 = 환경 설계자, AI = 실행자**

```
인간의 역할: SPEC 작성, 하네스 설계, 품질 기준 정의
AI의 역할:  설계된 환경(하네스) 안에서 실행
관계 패턴:  인간이 제약과 가이드를 설계하고 AI가 그 안에서 최대 성능 발휘
```

환경 설계가 중심인 모델. 초기 설계에 가장 많은 인간의 노력이 들어가지만, 이후 실행은 높은 신뢰도로 자동화된다.

### 4.3 공통 트렌드 분석

4개 방법론이 서로 다른 접근법을 취하면서도 공유하는 핵심 트렌드가 있다:

**1. SPEC/PRD 문서 중심화**

모든 방법론이 AI 실행 전 구조화된 문서(SPEC, PRD, prd.json)를 필수로 요구한다. 자연어 대화만으로 AI에게 복잡한 작업을 맡기던 시대에서, **명시적 계약서** 기반 협업으로 진화한 것이다.

**2. 품질 게이트의 다층화**

단순히 코드가 실행되는 것을 넘어, 각 단계에서 품질 기준을 통과해야 다음 단계로 진행할 수 있는 게이트 구조가 공통적으로 채택되고 있다. BMAD의 적대적 리뷰, Superpowers의 TDD + 증거 검증, MOAI ADK의 TRUST 5가 모두 이 트렌드의 표현이다.

**3. AI와 인간의 역할 명확한 분리**

초기 AI 코딩 도구들이 인간과 AI의 역할을 모호하게 섞어놓았다면, 이 방법론들은 **누가 무엇을 하는가**를 명확히 정의한다. 이는 AI 에이전트를 더 효과적으로 활용하는 동시에, 책임 소재를 분명히 하는 성숙한 접근이다.

#### Superpowers — 이미 설치됨, 이 보고서 작성에 직접 활용

Superpowers는 조사 시작 전부터 이미 설치되어 운영 중이었다. 이 보고서 자체가 Superpowers의 `writing-plans` → `executing-plans` 워크플로우를 따라 작성되었다.

```bash
# 설치 방법 (참고용)

cp -r superpowers/skills/* ~/.claude/skills/
```

**실습 소감**: 스킬 파일이 명확한 역할 지침을 제공하므로, 처음 사용하는 경우에도 방향을 잃지 않는다. `brainstorming` 스킬로 보고서 구조를 먼저 발산하고, `writing-plans`로 섹션별 계획을 세운 뒤 실행하는 흐름이 자연스럽게 연결됐다.

#### Ralph — ~/.claude/skills/ 복사

```bash
# Ralph 레포 클론
git clone https://github.com/snarktank/ralph.git
cd ralph

# 스킬 파일 복사
cp -r skills/* ~/.claude/skills/

# prd.json 템플릿 확인
cat templates/prd.json
```

**설치 난이도**: 낮음 (10분 이내)

**실습 소감**: prd.json 작성이 생각보다 명확한 사고를 요구한다. "원하는 것"을 구조화하는 과정 자체가 기획 훈련이 된다. 루프 실행 후 AI가 실제로 독립적으로 작업을 진행하는 것을 보며, 자율 실행의 강력함을 체감했다. 다만 그린필드 프로젝트가 아닌 기존 레포에 적용 시도했을 때 맥락 이해 실패를 경험했다.

## 7. 결론 및 개인 견해

### 7.1 AI 개발 방법론 공통 트렌드 3가지

이번 조사를 통해 4개 방법론이 각기 다른 접근법을 취하면서도, 업계가 수렴하는 세 가지 공통 트렌드를 확인할 수 있었다.

**트렌드 1: SPEC/PRD 문서 중심화**

모든 방법론이 자연어 대화가 아닌 구조화된 문서를 AI 협업의 출발점으로 삼는다. 이는 AI와의 협업이 성숙해지면서, "잘 정의된 문제"가 "잘 생성된 코드"보다 더 중요하다는 인식이 확산된 결과다. 앞으로의 AI 개발 방법론은 더 정교한 SPEC 언어와 템플릿 표준화를 향해 발전할 것으로 전망된다.

**트렌드 2: 품질 게이트의 다층화**

초기 AI 코딩 도구가 "코드를 생성해주는 도구"였다면, 이제는 "품질이 보증된 코드를 생성하는 시스템"으로 진화하고 있다. BMAD의 적대적 리뷰, Superpowers의 증거 기반 검증, MOAI ADK의 TRUST 5가 모두 이 방향성의 구현이다. 단일 품질 게이트에서 다층 품질 게이트로의 이동은 AI 에이전트에 대한 신뢰를 확보하기 위한 업계의 집단적 응답이다.

**트렌드 3: AI와 인간의 역할 명확한 분리**

"AI가 다 해준다"는 초기의 기대감이 식으면서, 성숙한 방법론들은 AI와 인간의 역할을 명확히 구분한다. 인간은 목표 설정, 품질 기준 정의, 최종 판단을 담당하고, AI는 그 틀 안에서 실행을 담당한다. 이 역할 분리는 AI 에이전트를 더 효과적으로 활용하는 동시에, 인간의 책임을 명확히 한다.

### 7.2 개인적으로 가장 주목하는 방법론

**Superpowers**를 가장 주목한다. 그 이유는 세 가지다.

첫째, **실증성**이다. 24건의 실패 사례에서 귀납적으로 도출한 원칙은 이론이 아니라 경험이다. AI 개발 방법론이 아직 초기 단계인 지금, "무엇이 실제로 실패하는가"를 알고 그것을 구조적으로 해결하려는 접근이 가장 신뢰할 만하다.

둘째, **확장성**이다. 스킬 파일 단위로 구성되어 있어 필요한 것만 선택 적용할 수 있다. BMAD나 MOAI ADK처럼 전체 시스템을 한 번에 도입하지 않아도 된다. 팀의 성숙도에 맞춰 점진적으로 적용 범위를 넓힐 수 있다.

셋째, **메타적 자기 참조성**이다. 이 보고서 자체가 Superpowers 방법론을 사용해 작성되었다. 방법론이 자기 자신의 작성에도 적용될 만큼 범용적이라는 것이 실질적 증명이다.

### 7.3 향후 발전 방향 전망

**단기 (1~2년)**: 방법론 간 상호운용성 증가. BMAD의 팀 시뮬레이션 + Superpowers의 TDD처럼, 서로 다른 방법론의 장점을 결합하는 하이브리드 접근이 확산될 것이다.

**중기 (3~5년)**: SPEC 언어의 표준화. 현재는 각 방법론이 자체 형식(prd.json, SPEC 파일)을 사용하지만, 방법론 간 호환되는 공통 SPEC 표준이 등장할 가능성이 높다. OpenAPI가 REST API를 표준화했듯이.

**장기 (5년+)**: AI 에이전트 자체가 방법론을 선택. 현재 인간이 방법론을 선택하고 AI가 그 안에서 실행하지만, 미래에는 AI 에이전트가 프로젝트 특성을 파악해 스스로 최적 방법론을 선택하는 메타 레이어가 등장할 것이다. MOAI ADK의 Orchestrator 에이전트가 그 방향의 초기 시도라고 볼 수 있다.
